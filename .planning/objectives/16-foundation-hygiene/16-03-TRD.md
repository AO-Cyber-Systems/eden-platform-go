# TRD 16-03 — Encryption Reconciliation

**Objective:** 16 (Platform Foundation Hygiene)
**Roadmap req:** R26.3
**Status:** planned

## Goal

`platform/encryption` is already production-grade (AES-256-GCM + HMAC-SHA256
blind index). Reach feature-parity with the patterns we know `aosentry/pkg/crypto`
and `eden-biz/internal/crypto` use, so future consumer migrations don't need to
add helpers back. No consumer migration in this objective.

## Tasks

1. Add convenience helpers (backward-compatible additions):
   - `EncryptString(s string) (string, error)` — base64-encoded ciphertext.
   - `DecryptString(s string) (string, error)` — inverse.
   - `BlindIndexLower(s string) string` — case-insensitive blind index for
     emails / usernames (matches Eden-Biz pattern).
2. Versioned ciphertext envelope:
   - New helpers `EncryptStringV1` / `DecryptString` accept and produce a
     `v1:<base64>` prefix. `DecryptString` accepts both prefixed (v1) and
     unprefixed (legacy raw nonce+ciphertext) for migration safety.
3. Key loading helpers:
   - `KeyFromHex(s string) ([]byte, error)`
   - `KeyFromBase64(s string) ([]byte, error)`
   - `KeyFromEnv(name string) ([]byte, error)` — reads env var, accepts hex
     or base64 (auto-detected by length / valid alphabet), validates 32-byte
     length.
4. Document migration in `platform/encryption/README.md`:
   - Key formats accepted
   - At-rest data compatibility: existing AES-GCM payloads remain decryptable;
     new payloads can opt into the `v1:` prefix.
5. Tests: cover all new exports.
