# TRD 16-02 — Observability Consolidation

**Objective:** 16 (Platform Foundation Hygiene)
**Roadmap req:** R26.2
**Status:** planned

## Goal

Consolidate the slog/sentry/metrics/interceptor helpers under
`platform/observability` so consumers have ONE place to wire their service-level
observability. The existing `platform/errortrack` package (Sentry transport with
PII scrubber + slog handler) stays where it is — observability re-exports a
boot-time façade that wires errortrack + slog + metrics together.

## Tasks

1. Add a boot helper in `platform/observability` that consumers call once at
   `main()`:
   - `Setup(cfg Config) (Shutdown, error)` returns a flush/close function.
   - Wires: errortrack.Init (Sentry), slog default handler (JSON or text),
     errortrack.SlogHandler combined via MultiHandler, *Metrics.
   - `Config` covers: ServiceName, Environment, Release, SentryDSN,
     LogLevel, LogFormat, SampleRate.
2. Add trace-context propagation on the existing interceptor:
   - When the request context carries a `slog.Attr` named `trace_id`/`span_id`
     (set by an upstream OTel middleware), attach it to the slog log line for
     each RPC.
   - No new OTel SDK dependency — the helper just plumbs values it finds.
3. Document the canonical observability surface in
   `platform/observability/README.md`:
   - When to call `Setup`
   - How to add product-specific metrics on top of `*Metrics.Registry`
   - How errortrack relates (single Sentry transport for the whole portfolio)
4. Tests: cover `Setup` (no-op DSN path + custom DSN with capturing transport),
   trace-id attachment in interceptor.

## Files
